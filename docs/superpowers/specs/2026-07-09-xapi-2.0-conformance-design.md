# xAPI 2.0 Conformance — Design

**Date:** 2026-07-09
**Status:** Draft for review

## Goal

Make the LRS formally conformant with xAPI 2.0 (IEEE 9274.1.1-2023) while keeping full
xAPI 1.0.3 conformance for existing clients.

**Success criteria:**

1. The official ADL LRS Conformance Test Suite passes at `mustPassAll` level in **both**
   the `v1_0_3` and `v2_0` batteries, with **zero failing tests and zero pending tests
   outside a justified allowlist**. The allowlist must be empty unless an entry documents
   a defect in the suite itself (with an upstream issue link); it never covers gaps in
   our implementation.
2. Both batteries run in CI against both database drivers (`pg`, `pglite`).
3. Existing 1.0.3 clients are unaffected: no endpoint, header, or validation behavior
   changes for requests carrying a `1.0.x` version header.

## Current state

- The LRS is 1.0.3-conformant. Version negotiation lives in `src/app.ts` (middleware,
  ~line 287), statement validation in `src/xapi/statement-schema.ts` +
  `statement-validator.ts`, and `/xapi/about` advertises `['1.0.3']`.
- The conformance harness (`test/conformance/run-adl-suite.ts` +
  `adl-conformance.test.ts`) runs `adl-lrs-conformance-tests@1.2.3` from npm. That
  package is a **stale third-party fork** (TryxAPI, last published June 2022) that ships
  only `v1_0_2`/`v1_0_3` test batteries. It cannot test 2.0.
- The official suite, `adlnet/lrs-conformance-test-suite` (master, version `2.0.0-1`,
  actively maintained), contains a full `v2_0` battery but **is not published to npm**.
- Statements are stored as complete JSON in `xapi_statement.payload`. xAPI 2.0's data
  model additions fit inside that payload, so **no database schema changes are needed**
  (the schema is frozen; this design respects that).

## Architecture decision

**Dual-version, single pipeline, version-aware validation** (chosen over 2.0-only and
over a canonical-model/translation layer):

- A version-negotiation middleware determines the request's xAPI version (`1.0.x` or
  `2.0.x`) and stores a normalized value (`'1.0.3' | '2.0.0'`) in the Hono context.
- One statement pipeline and one storage format. Validation is parameterized by the
  negotiated version.
- Statements are stored as received (after spec-mandated normalizations) and served
  as-is to readers of either version. Rationale: the immutability rules effectively
  require serving statements substantially as received; a translation layer would fight
  them for little gain.

**Cross-version read policy (explicit):** a statement stored via the 2.0 API that
contains 2.0-only fields (e.g. `contextAgents`) is returned unmodified to a 1.0 reader.
The spec requires clients to tolerate unknown properties, and the LRS must not modify
statements based on version (XAPI-00330).

## Phase 1 — Conformance harness upgrade (independently landable)

Land this before any server changes. Exit criterion: the `v1_0_3` battery from the
**official** suite passes in CI. Any regressions the fresher battery surfaces are fixed
as part of this phase.

### 1a. Swap the suite dependency

- Replace `adl-lrs-conformance-tests@1.2.3` (npm) with a git dependency on
  `adlnet/lrs-conformance-test-suite`, **pinned to a commit SHA** for reproducibility.
- Verify the runner API in `run-adl-suite.ts` against the new package: the constructor
  now accepts an `xapiVersion` flag alongside `directory`; confirm the message-based
  lifecycle (`suite start` / `end`), `getCleanRecord()`, and `summary` fields still
  behave as the wrapper expects, and adapt if not.

### 1b. Parameterize the runner

- `runConformanceSuite({ xapiVersion: '1.0.3' | '2.0.0', ... })` selects the battery
  (equivalently `directory: ['v1_0_3'] | ['v2_0']` — the suite accepts either, not both).
- The CLI entrypoint takes the version as an argument.

### 1c. Harden the vitest wrapper

Current holes, each closed in `adl-conformance.test.ts`:

| Hole                                                                                            | Fix                                                                                                                                     |
| ----------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| Unmatched section titles are silently skipped (`findSection` returns `undefined` → test passes) | Fail the test when an expected section is missing; also fail a guard test if the suite reports sections not in our map                  |
| Pending/cancelled tests pass in vitest (only the CLI checks them)                               | Assert `pending === 0`, with an explicit, documented allowlist for known-pending tests (empty to start)                                 |
| No floor on total test count — a shrunken battery goes green                                    | Assert `result.total` ≥ a per-battery floor constant, set just below the observed count                                                 |
| `SPEC_SECTIONS` map is 1.0.3-only                                                               | Per-battery section maps; the `v2_0` battery uses IEEE-style numbering (e.g. `4.1.6.1 Statement Resource`) alongside carried-over names |

Keep: `mustPassAll` level, `expect.soft` per-requirement diagnostics, grep filtering.

### 1d. CI

- Matrix: `{version: [1.0.3, 2.0.0]} × {driver: [pg, pglite]}` = 4 parallel conformance
  jobs. (The 2.0 jobs are expected red until Phase 2 completes; they run non-blocking —
  e.g. `continue-on-error` — until Phase 2 flips them to required.)
- Auth surface under test remains Basic Auth only (OAuth1 is intentionally unsupported).

## Phase 2 — 2.0 server changes (driven red→green by the v2_0 battery)

### 2a. Version negotiation (`src/app.ts`)

- Accept `X-Experience-API-Version` values of `1.0` / `1.0.x` **and** `2.0` / `2.0.x`;
  reject anything else with 400. Error responses must still carry the
  `X-Experience-API-Version` response header (the battery checks this).
- Respond with the latest patch of the negotiated major: `1.0.3` for 1.0.x requests,
  `2.0.0` for 2.0.x requests.
- `/about` (and `/stream`): header remains optional; when present it must be a supported
  version. When present, the response header echoes that major's latest patch; when
  absent, respond with the latest supported version (`2.0.0`) — currently hardcoded to
  `1.0.3` at `src/app.ts:296`. Verify this choice against both batteries.
- CORS `allowHeaders`/`exposeHeaders` unchanged (header names are version-agnostic).

### 2b. Alternate request syntax (`?method=POST`)

Removed in xAPI 2.0; retained for 1.0 clients. The override middleware runs before
version negotiation, so gate it on the request's **effective version**: the
`X-Experience-API-Version` form-body field if present, else the request header (body
fields are optional in 1.0.3's alternate syntax). Honor the rewrite only when the
effective version is `1.0.x`. The v2_0 battery does not test for rejection,
so no new rejection behavior is added — the syntax is simply not part of the 2.0 surface.

### 2c. About resource (`src/routes/about.ts`)

- `version: ['1.0.3', '2.0.0']`. The battery requires the exact string `"2.0.0"` to be
  present (XAPI-00317).

### 2d. Version-parameterized statement validation

`statement-schema.ts` / `statement-validator.ts` become version-aware:

- **`context.contextAgents` / `context.contextGroups`** (2.0 only): arrays of objects
  with `objectType: 'contextAgent' | 'contextGroup'`, an `agent`/`group` member, and
  optional `relevantTypes` (non-empty array of IRIs). Rejected for 1.0 requests
  (unknown-property rule, matching current behavior).
- **`version` property:** `1.0.x` for 1.0 requests (current rule);
  `2.0.x` allowed for 2.0 requests. Stored as received — no up/down-conversion.
- **Timestamps (2.0):** RFC 3339 accepted (relaxed from strict ISO 8601 where they
  differ); a non-UTC offset is **converted to UTC** on write instead of rejected;
  invalid formats still rejected. The `-00:00` offset is rejected per spec. 1.0
  behavior unchanged.
- Everything else (IFIs, verbs, interaction activities, attachments metadata) is
  unchanged; the battery will confirm.

### 2e. Response headers

- Add `Last-Modified` to responses where the v2_0 battery requires it (new 2.0 LRS
  requirement; exact resource list driven by the battery — statements and document
  resources).
- `X-Experience-API-Consistent-Through` behavior unchanged.

### 2f. Statement equality and voiding

- `statement-compare.ts` (statement-ID conflict detection / immutability comparison)
  must include `contextAgents`/`contextGroups` in its comparison logic.
- Voiding semantics are unchanged in 2.0.

### 2g. Signed statements

- JWS mechanics unchanged; verify the v2_0 battery's expectations (e.g. version property
  inside the signed copy) and adjust `signature.ts` only if the battery demands it.

### 2h. Document resources, concurrency, error codes

- Known-in-advance change: xAPI 2.0 extends concurrency requirements (ETag /
  `If-Match` / `If-None-Match`) to the **State resource**, not just the profile
  resources. Plan for it explicitly.
- Beyond that and `Last-Modified` (2e), no known behavioral changes; treat as
  audit-and-tighten driven by battery failures rather than speculative rewrites.

## Error handling

- Version rejection: 400 with the existing JSON error shape, version header present.
- Validation failures: existing 400 pathways, now version-aware. No new error formats.

## Testing strategy

- **Conformance (the bar):** both batteries × both drivers in CI, `mustPassAll`, zero
  failing, zero pending, section-map and total-count guards active.
- **Unit tests:** version middleware negotiation table; 2.0 schema additions
  (contextAgents/contextGroups shapes, version regexes, timestamp normalization and
  `-00:00` rejection); alternate-syntax gating.
- **Integration tests:** cross-version read (store via 2.0 with contextAgents, read via
  1.0, payload returned as-is); `/about` contents; response headers per version.

## Risks and mitigations

| Risk                                                                             | Mitigation                                                                                                                       |
| -------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Runner API drift between the 2022 fork and official master breaks the harness    | Phase 1a verifies against the pinned SHA before anything else lands                                                              |
| The official `v1_0_3` battery (3 years newer than the fork) surfaces regressions | Treated as in-scope Phase 1 fixes; they block Phase 1 exit                                                                       |
| `v2_0` battery immaturity (bugs in the suite itself)                             | SHA pinning isolates us from upstream churn; suite bugs get reported upstream and documented in the allowlist with justification |
| Conformance CI time doubles                                                      | Parallel matrix jobs keep wall-clock flat                                                                                        |

## Out of scope

- OAuth1 authentication (suite supports it; we deliberately don't)
- xAPI 0.9 / 0.95 legacy version support
- Any database schema change (schema is frozen)
- Canonical-model translation between versions (rejected as Option C)
- cmi5 or profile-level conformance
