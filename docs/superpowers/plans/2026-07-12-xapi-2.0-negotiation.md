# xAPI 2.0 Phase 2A: Version Negotiation Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the LRS negotiate xAPI `2.0.x` requests (accept the header, advertise `2.0.0` at `/about`, gate the removed alternate-request-syntax to 1.0 only) so the ADL v2_0 conformance battery can bootstrap — then re-run it to record the real per-requirement failure baseline that drives the rest of Phase 2.

**Architecture:** The version-header middleware in `src/app.ts` becomes version-aware: it accepts `1.0.x` **and** `2.0.x`, stores the negotiated version (`'1.0.3' | '2.0.0'`) in the Hono context, and echoes the negotiated major's latest patch in the response header. `/about` advertises both versions. The alternate-syntax middleware (which runs before negotiation) is gated to the request's effective 1.0.x version. No statement-level 2.0 behavior changes here — that's Phase 2B, planned from the baseline this plan produces. The 1.0.3 battery staying green (37/37) is a hard regression gate on every task.

**Tech Stack:** TypeScript/Hono, vitest, the ADL LRS conformance suite (v1_0_3 + v2_0 batteries).

**Spec:** `docs/superpowers/specs/2026-07-09-xapi-2.0-conformance-design.md` — sections **2a** (version negotiation), **2b** (alternate request syntax), **2c** (about resource). Sections 2d–2h are explicitly **out of scope** for this plan.

**Prior baseline:** `docs/superpowers/plans/2026-07-09-xapi-2.0-red-baseline.md` — records that the v2_0 battery currently aborts at its root `setTimeMargin` `before` hook because the server 400s the hook's `2.0.0` version header. This plan's final task supersedes that doc with the real per-section failures.

**Branch:** stacked on `lrsql-schema-compat` (#72). When #72 merges to `main`, re-target this branch's PR to `main` and rebase.

**Verified facts this plan relies on** (checked against the current code + the pinned suite):

- Version-header middleware is `src/app.ts:287-310`. It currently hard-rejects any header not starting with `1.0` (400) and hard-codes `X-Experience-API-Version: 1.0.3` on every response. This is the exact gate the v2_0 battery's bootstrap hits.
- The alternate-syntax middleware is `src/app.ts:215-281` and runs **before** the version middleware. `ALTERNATE_HEADER_FIELDS` (app.ts:36) includes `X-Experience-API-Version`, so the effective version can be read from the form body when present.
- `/about` is `src/routes/about.ts`; the handler returns `{ version: ['1.0.3'] }`. The middleware's `/about`/`/stream` branch (app.ts:292-298) makes the header optional but currently rejects non-`1.0` and echoes `1.0.3`.
- **The v2_0 battery's bootstrap statement is version-less** (`test/v2_0/templates/statements/default.json` = actor/verb/object only; no `version` field), and the `2.0.0` lives only in the request header (`helper.js addHeaderXapiVersion` → `process.env.XAPI_VERSION`). So accepting the `2.0.x` header is sufficient to unblock bootstrap — statement validation is untouched.
- `HonoEnv` (`src/hono-env.ts`) is where a typed `xapiVersion` context var is declared.
- CORS `allowHeaders`/`exposeHeaders` (app.ts:197-198) already list `X-Experience-API-Version` — no change needed.
- The conformance harness reads `CONFORMANCE_XAPI_VERSION`; `pnpm test:conformance` runs 1.0.3, `pnpm test:conformance:2.0` runs 2.0. `spec-sections.ts` holds the section maps + `TOTAL_FLOOR`.

**Local run note:** `DATABASE_DRIVER=pglite` for everything (no external Postgres needed). `pnpm run fmt` before every commit (oxfmt/oxlint pre-commit hooks bounce unformatted files).

- [ ] **Step 1.3: Implement the negotiation middleware**

Replace `src/app.ts:287-310` with version-aware logic:

```typescript
  // --------------------------------------------------------------------------
  // xAPI Version Negotiation (§3.3)
  // Accept 1.0.x and 2.0.x. Store the negotiated version in context and echo
  // the negotiated major's latest patch in the response header. /about and
  // /stream keep the header optional.
  // --------------------------------------------------------------------------

  const negotiate = (version: string | undefined): '1.0.3' | '2.0.0' | null => {
    if (version === undefined) return null;
    if (/^1\.0(\.\d+)?$/.test(version)) return '1.0.3';
    if (/^2\.0(\.\d+)?$/.test(version)) return '2.0.0';
    return null; // unsupported
  };

  app.use('/xapi/*', async (c, next) => {
    const version = c.req.header('x-experience-api-version');
    const path = c.req.path;
    const negotiated = negotiate(version);

    // /about and /stream: header optional; if present it must be supported.
    if (path === '/xapi/about' || path === '/xapi/stream') {
      if (version !== undefined && negotiated === null) {
        c.header('X-Experience-API-Version', '2.0.0');
        return c.json({ error: `Unsupported xAPI version: ${version}` }, 400);
      }
      const resolved = negotiated ?? '2.0.0'; // absent → latest supported
      c.set('xapiVersion', resolved);
      c.header('X-Experience-API-Version', resolved);
      return next();
    }

    // All other xAPI routes: header required and must be supported.
    if (version === undefined) {
      c.header('X-Experience-API-Version', '2.0.0');
      return c.json({ error: 'X-Experience-API-Version header is required' }, 400);
    }
    if (negotiated === null) {
      c.header('X-Experience-API-Version', '2.0.0');
      return c.json({ error: `Unsupported xAPI version: ${version}` }, 400);
    }
    c.set('xapiVersion', negotiated);
    c.header('X-Experience-API-Version', negotiated);
    return next();
  });
```

- [ ] **Step 3.2: Implement the effective-version gate**

At the top of the alternate-syntax middleware (`src/app.ts:215`, after the method/content-type guards but before the rewrite), compute the effective version and bail out of the rewrite for non-1.0 requests. The effective version is the form-body `X-Experience-API-Version` field if present, else the request header:

```typescript
    // Alternate request syntax was removed in xAPI 2.0. Gate on the request's
    // effective version (form-body field if present, else the header) and only
    // rewrite for 1.0.x. For 2.0.x, fall through — the version middleware and
    // normal POST handling apply (a POST with ?method= is just a POST in 2.0).
    const bodyVersion = formData['X-Experience-API-Version'];
    const headerVersion = c.req.header('x-experience-api-version');
    const effective = bodyVersion ?? headerVersion;
    if (effective !== undefined && !/^1\.0(\.\d+)?$/.test(effective)) {
      // Not 1.0.x: do not rewrite. Rebuild the original request minus ?method
      // is unnecessary — just continue to the next middleware as a POST.
      return next();
    }
```

Note: `formData` is already built at `app.ts:224-230` (after the method/content-type guards, before the URL rebuild) from the single `c.req.text()` read at line 225. Insert the effective-version check immediately after `formData` is constructed (~line 230), before the URL rebuild — **no move is required, and do not add a second `c.req.text()`**. Keep the existing 1.0 rewrite intact for `effective` that is 1.0.x or absent-with-1.0-header.
