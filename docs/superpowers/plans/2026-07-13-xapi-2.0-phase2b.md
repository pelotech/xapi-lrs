# xAPI 2.0 Phase 2B Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the 10 remaining v2_0 conformance failures — accept `context.contextAgents`/`contextGroups` and the `2.0.x` statement `version` property (2.0 requests only), decompose the new context actors, add the `Last-Modified` header to statement GETs, reject same-`id` batches, and enforce State-resource ETag concurrency — driving the v2_0 ADL battery to fully green while the 1.0.3 battery stays 37/37.

**Architecture:** Validation becomes version-aware by threading the already-negotiated `c.var.xapiVersion` into `validateStatement(input, version)`, which selects a version-parameterized schema (the 2.0 schema adds the two context arrays and widens the `version` regex; the 1.0 schema is byte-for-byte today's behavior). Decomposition gains a contextAgents/contextGroups walk using the `ContextAgent`/`ContextGroup`/`Sub*` enum values the lrsql schema already carries. The other three fixes reuse existing patterns: `Last-Modified` mirrors the document-resource header code, batch-dup is a set-size guard, and State concurrency flips one flag on the shared `checkConcurrencyHeaders` helper (version- and existence-gated).

**Tech Stack:** TypeScript/Hono, Zod, vitest, the ADL v2_0 conformance battery.

**Spec:** `docs/superpowers/specs/2026-07-09-xapi-2.0-conformance-design.md` — sections 2d, 2e, 2f, 2h.
**Baseline (the 10 failures):** `docs/superpowers/plans/2026-07-12-xapi-2.0-negotiation-baseline.md`.
**Branch:** stacked on `xapi-2.0-negotiation` (which is stacked on `lrsql-schema-compat` #72). Re-target/rebase as the stack merges.

- [ ] **Step 1.2: Version-parameterized schema in statement-schema.ts**

Add the context-actor/group schemas (reusing `agentSchema`, `groupSchema`, `iri` already in the file):

```typescript
const contextAgentSchema = z
  .object({
    objectType: z.literal('contextAgent'),
    agent: agentSchema,
    relevantTypes: z.array(iri).min(1).optional(),
  })
  .strict();

const contextGroupSchema = z
  .object({
    objectType: z.literal('contextGroup'),
    group: groupSchema,
    relevantTypes: z.array(iri).min(1).optional(),
  })
  .strict();
```

Make the schema construction version-parameterized. The clean shape: a factory that builds the `contextSchema` and the top-level `version` field per version, then the statement + substatement schemas. Concretely — parameterize the two things that differ:

1. `contextSchema`: for 2.0, add `contextAgents: z.array(contextAgentSchema).optional()` and `contextGroups: z.array(contextGroupSchema).optional()` to the `.object({...})` before `.strict()`. For 1.0, exactly today's fields. (Both stay `.strict()` — 2.0 just permits the two new keys.)
2. The `version` regex: 1.0 → `/^1\.0(\.\d+)?$/` (unchanged); 2.0 → `/^(1\.0|2\.0)(\.\d+)?$/` (accept either major, since a 2.0 request may legitimately carry a 1.0.x-authored statement — the version is stored as received).

Build and memoize both `statementInputSchema` variants at module load (e.g. `buildStatementInputSchema('1.0.3')` / `('2.0.0')`, or two constants). Export a selector: `export function statementSchemaFor(version: XapiVersion) { return version === '2.0.0' ? schema20 : schema10; }`. Import `XapiVersion` from `./versions.ts`.

**CRITICAL — the full rebuild chain and the superRefine.** `contextSchema` is nested three levels deep: `contextSchema` → `subStatementSchema` (~line 328) → `objectSchema` (~lines 336-342) → `statementInputSchema` (~line 363). So parameterizing `contextSchema` forces the factory to rebuild `subStatementSchema`, `objectSchema`, AND `statementInputSchema` per version — not just "the statement + substatement schemas." **Both `subStatementSchema` and `statementInputSchema` carry `.superRefine(checkContextActivityOnly)`** (the revision/platform-only-on-Activity rule). The factory MUST re-attach that same `.superRefine(checkContextActivityOnly)` to both rebuilt schemas, or the 1.0 "byte-for-byte preserved / 37/37" guarantee silently regresses (that rule would vanish). Structure the factory as one function that takes the version, builds the version's `contextSchema` + `version` regex, threads them through the full chain, and re-attaches the superRefine — parameterize only what differs, but rebuild the whole dependent chain. After implementing, diff the 1.0 schema's behavior against pre-change by running the full 1.0.3 battery (Step 1.4) — it must stay exactly 37/37.
