## Bootstrap failure

The 2.0 battery never reaches any conformance test. The suite's `setTimeMargin`
helper is a root-level Mocha `before` hook (declared outside any `describe()`,
in `test/v2_0/Data2.2-FormattingRequirements.js`) that POSTs one statement
before any section runs, to measure clock skew between the suite and the LRS.
That POST carries `X-Experience-API-Version: 2.0.0`. The server's version
middleware (`src/app.ts`, ~lines 288–307) rejects any version not starting with
`1.0` with `400 Bad Request` on all xAPI routes, so the hook throws and Mocha
aborts the run before entering any of the 34 top-level sections.

What this does to the recorded data:

- Exactly one failure: the root hook itself, with an empty title and no
  requirement ID (`Error: Expected response status code to be 200 got 400`,
  thrown from the suite's `helper.js:554`).
- 1434 pending tests with no per-test data anywhere in the structured output.
  The runner's clean-log tree is a single degenerate node (empty title, zero
  children); the 1434 exists only as an aggregate counter in the runner
  summary. No requirement IDs, section names, or per-test detail exist for
  any of them.
- No `suite start` event with a section title was emitted; no section from
  `SPEC_SECTIONS_2_0_0` appeared in the run at all.

This also explains the earlier grep-limited 2.0 smoke run ("About Resource":
Total 15, Passing 0, Failing 1, Pending 14). Its single failure was this same
bootstrap hook, not the about-version test as first assumed.

## Consequences for Phase 2 planning

- No per-requirement red list can exist until version negotiation (design spec
  section 2a) lands; the battery cannot get past its own setup traffic.
- Phase 2's first task must therefore be version negotiation plus `/about`
  version advertising (spec sections 2a + 2c). After that lands, re-run the
  battery and re-record this document with the real per-section failure table.
- The v2_0 map's `mayBeAbsent` flag on Headers (`(Communication 1.2)`) is
  unverified until that re-run; no section appeared in this run, so it carries
  no evidence either way.
- The known-in-advance gap list from the design spec (contextAgents /
  contextGroups, Last-Modified headers, timestamp conversion, statement
  version property) remains the expected failure surface, unconfirmed by test
  evidence.

## Notes for Phase 2 planning

- CI visibility: the 2.0 matrix jobs run with `continue-on-error`, and
  GitHub's aggregate PR status does not distinguish "passed" from "failed but
  tolerated". While the 2.0 battery is expected to fail, reviewers must check
  the per-job status in the Actions tab, not the green aggregate check.
  (Remove `continue-on-error` when the 2.0 battery goes green.)
- `validateSince` (`src/helpers/agent.ts`) validates `since`/`until`
  statement-query parameters via `new Date()`, which accepts `-00:00` and
  looser forms. The battery doesn't test this, but Phase 2 should consider
  aligning query-parameter timestamp validation with the statement timestamp
  rules.
