## Failures by section

Grouped by the top-level `describe()` titles as reported by the runner (the
`spec-sections.ts` section labels are shown in brackets). Each group maps to a
design-spec section of `docs/superpowers/specs/2026-07-09-xapi-2.0-conformance-design.md`.

### Context Property Requirements (Data 2.4.6) — 4 failures → design **2d** (contextAgents / contextGroups)

The server rejects the two new 2.0 context properties with `400` (unknown-property
rule still applies for 2.0 requests; these must be accepted). Fires for both the
top-level statement and the sub-statement:

| #   | Leaf test                                                                                         | Error                                           |
| --- | ------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| 1   | Statement with `contextAgents` property has an array of valid `contextAgent` Objects              | Expected response status code to be 200 got 400 |
| 2   | Statement substatement with `contextAgents` property has an array of valid `contextAgent` Objects | Expected response status code to be 200 got 400 |
| 3   | Statement with `contextGroups` property has an array of valid `contextGroup` Objects              | Expected response status code to be 200 got 400 |
| 4   | Statement substatement with `contextGroups` property has an array of valid `contextGroup` Objects | Expected response status code to be 200 got 400 |

Design 2d: `context.contextAgents` / `context.contextGroups` are arrays of objects
with `objectType: 'contextAgent' | 'contextGroup'`, an `agent`/`group` member, and
optional non-empty `relevantTypes` IRI array — accepted for 2.0 requests, still
rejected for 1.0. Applies at both statement and sub-statement scope.

### Version Property Requirements (Data 2.4.10) — 1 failure → design **2d** (version property)

| #   | Requirement                   | Leaf test                                                                                                               | Error                                           |
| --- | ----------------------------- | ----------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| 5   | XAPI-00332 (`Data.md#2.4.10`) | Statements returned by an LRS MUST retain the version property they are accepted with (Format, Data 2.4.10, XAPI-00332) | Expected response status code to be 200 got 400 |

The server rejects a statement carrying a `2.0.x` `version` property. Design 2d:
`2.0.x` allowed for 2.0 requests, stored as received (no up/down-conversion);
`1.0.x` rule unchanged for 1.0 requests.

### Statement Resource Requirements (Communication 2.1) — 2 failures → design **2e** (Last-Modified) + **2f** (statement equality / batch)

| #   | Leaf test                                                                                    | Error                                                                | Design |
| --- | -------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- | ------ |
| 6   | should include a Last-Modified header which matches the "stored" Timestamp of the statement. | The Last-Modified header could not be parsed — received: `undefined` | **2e** |
| 7   | should reject a batch of two or more statements where the same ID is used more than once.    | `TypeError: Cannot read properties of undefined (reading 'status')`  | **2f** |

- **#6 (2e):** the statement GET response carries no `Last-Modified` header (new
  2.0 LRS requirement; add to statement — and per design 2e, document — resources).
- **#7 (2f):** the suite POSTs a two-statement batch whose members share one `id`
  and expects `400`. The failure is a suite-side `TypeError` reading `res.status`
  / `err.response.status` (suite `4.1.6.1-Statement-Resource.js:240`,
  `xapiRequests.sendStatementPromise`) — the promise wrapper could not read a
  status off the response, so the LRS's batch-duplicate-ID response does not match
  the shape the suite's client expects. Phase 2B must confirm the LRS rejects a
  same-`id` batch with a plain `400` (and a body/response the suite client can
  read); this is statement-equality / batch-validation territory (2f). Needs a
  short investigation to distinguish a genuine batch-dup gap from suite fragility.

### (4.1.4) Concurrency — 3 failures → design **2h** (State-resource ETag concurrency)

xAPI 2.0 extends the ETag / `If-Match` / `If-None-Match` concurrency requirements
to the State resource (design 2h). The LRS currently applies no precondition
enforcement on this path, so the conflicting write is accepted:

| #   | Leaf test                                     | Error                                              |
| --- | --------------------------------------------- | -------------------------------------------------- |
| 8   | Return 409 conflict                           | expected 204 to equal 409                          |
| 9   | Return error message explaining the situation | expected '' not to be empty                        |
| 10  | Do not modify the resource                    | expected `{ Object }` to deeply equal `{ Object }` |

All three are facets of the same gap: on a concurrency conflict the LRS must
return `409`, include an explanatory error body, and leave the stored resource
unmodified. Currently it returns `204` and overwrites the resource. Design 2h.
